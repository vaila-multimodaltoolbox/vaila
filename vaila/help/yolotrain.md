# yolotrain

## Module Information

- **Category:** ML
- **File:** `vaila/yolotrain.py`
- **Version:** 0.3.67
- **Updated:** 2026-06-29
- **GUI Interface:** Yes
- **CLI Interface:** Yes

## Description

YOLO26-first trainer for datasets generated by `vaila/getpixelvideo.py`, `vaila/vaila_sam.py` workflows, and existing Ultralytics YOLO datasets. It detects dataset task from `data.yaml` and selects the correct YOLO26 model by default.

> **Direct answer for player tracking:** use a `detect` dataset with one bounding
> box per player. `auto` only inspects the dataset; it is not a tracker. Use
> `pose` only when each player has real body-joint annotations.

## Task Guide

- **auto**: reads `data.yaml` and labels, then chooses detect/pose/segment.
- **detect**: YOLO bbox labels from getpixelvideo bbox export or the legacy CSV helper. Auto model: `yolo26m.pt`.
- **pose**: YOLO-pose labels with `kpt_shape` in `data.yaml`, including getpixelvideo F9 exports and FIFA keypoint datasets. Auto model: `yolo26m-pose.pt`.
- **segment**: polygon labels. Auto model: `yolo26m-seg.pt`.

For individual player tracking, start with **detect**. The detector produces one
box per player; BoT-SORT or ByteTrack assigns IDs during inference.

The CLI rejects known model/task mismatches early. Example: a dataset with `kpt_shape` is YOLO-pose, so `--model yolo26x.pt` is rejected and the suggested replacement is `--model yolo26x-pose.pt`.

## GUI

Open the main vailá window, choose **YOLO + SAM**, then **Train YOLOv26**. The trainer runs in a separate process from the main GUI.

Recommended flow:

1. Generate/export the dataset in `getpixelvideo.py` or from a SAM workflow loaded into `getpixelvideo.py`.
2. Select the dataset folder, its parent folder, or its `data.yaml`. If a parent folder contains one or more `data.yaml` files, the GUI prints all candidates in the terminal and selects the newest one by default.
3. Keep **Task** and **Model** as `auto` unless intentionally overriding.
4. To **fine-tune** or **continue training**, click **Browse Weights** (or type a path in **Model**) and pick any `.pt` file — e.g. `runs/yolo_training/weights/last.pt` or weights stored outside the dataset folder. Only catalog names like `yolo26m.pt` download into `vaila/models/`.
5. Set **Maximum epochs**, batch, image size, device, and start training.

The terminal prints resolved `train`/`val` paths, image counts, label counts, detected task, class names, and full tracebacks for YAML or training errors. The purple CSV helper remains only for older marker CSV files that do not already have a YOLO dataset.

The GUI now displays a colored plain-language task explanation. A suspicious
single-object pose dataset with many generic keypoints shows a confirmation
dialog before training. Choose **No** for the players-as-keypoints export and
rebuild it as detection.

**Single Help button (v0.3.65):** the GUI has one **Help** button (instead of
separate Help / Open HTML / Diagnostics / Model Guide / Task Guide buttons). It
opens an in-app guide window that also contains **Open full guide in browser
(HTML)** and **Print diagnostics + CLI command** buttons. The same quick guide
is printed to the terminal.

**STOP in the terminal (v0.3.65):** when a dataset looks like a
players-as-keypoints mis-export, the suspicious-pose gate now prints a clear
`STOP` block with the `sam_to_yolo` repair command to the terminal, not only the
GUI dialog.

**Temporal split:** datasets exported from `getpixelvideo` / `sam_to_yolo`
default to a chronological train/val/test split (see those tools' help). This
avoids the inflated mAP you get when a random split leaks near-duplicate video
frames between train and val.

## Dataset Guide

| Dataset | One label row means | Quick sign |
|---|---|---|
| detect | one player box | five values; no `kpt_shape` |
| pose | one object box plus all keypoints belonging to that object | `kpt_shape: [N, 3]` |
| segment | one object polygon/mask | many polygon coordinates |

If a frame contains 18 players, a detection label normally contains 18 rows.
For real person pose, it also contains one row per player, with that player's
body joints after the box values.

## Model Guide

| Size | Recommended use | Trade-off |
|---|---|---|
| `n` / `s` | quick tests, weaker GPU | fastest, usually less accurate |
| `m` | first serious run | balanced speed, VRAM, accuracy |
| `l` / `x` | later experiments | slower/heavier; cannot repair weak labels |

Task suffix must match: `yolo26m.pt` (detect), `yolo26m-pose.pt` (pose),
`yolo26m-seg.pt` (segment). For distant soccer players, start with
`yolo26m.pt`, `imgsz=1280`, and batch 4, 8, or `-1` auto.

## Maximum Epochs

The GUI field accepts 1–10000. `epochs` is a maximum; early stopping may finish
sooner when validation stops improving.

- 20: smoke test for labels, memory, and loss behavior.
- 100: first baseline.
- 200+: only while validation metrics still improve.

More epochs cannot fix the wrong task or incorrect labels.

### How to know training is running (v0.3.57)

After **Start Training**, watch the **same terminal** that launched vailá (the trainer is a subprocess with unbuffered `-u`):

1. **Phase banners** — `>> yolotrain: Preparing dataset cache...` before the first epoch (large datasets with 10k+ images can sit here for several minutes).
2. **Ultralytics tqdm bars** on stderr — batch progress within each epoch.
3. **Epoch summaries** — `>> yolotrain: Epoch 3/100 complete | train_loss=... | mAP50=...` after each epoch.
4. **GUI console** — same epoch lines and banners (tqdm `\r` bars stay in the terminal only).

Weights and plots land under `<dataset>/runs/yolo_training/weights/` (`best.pt`, `last.pt`).

### Class name `object` vs `person` (retrain / detect)

The name in `data.yaml` (`names: ['object']` or `['person']`) is **metadata** for labels and inference display — it does not change learned weights by itself.

- **Pose dataset** (`kpt_shape: [N, 3]`, task `pose`): keep **Task=auto**, model `yolo26m-pose.pt`. Class name is cosmetic.
- **Person detection** (bbox-only): export a **detect** dataset (no keypoints), set `names: ['person']`, **Task=detect**, model `yolo26m.pt`.
- **Continue training** an existing run: click **Browse Weights** or set **Model** to your `best.pt` / `last.pt` path (absolute or relative, e.g. `runs/yolo_training/weights/last.pt`). The trainer uses that file directly instead of looking under `vaila/models/`.

### Player tracking: correct SAM → YOLO workflow

To **detect/track N people or objects** you need a **detection** dataset: one
bounding box per instance per frame, a single class (e.g. `person`), and **no
`kpt_shape`**. The number of people is **not** the number of classes — the
*tracker* (BoT-SORT/ByteTrack) assigns IDs `1..N` at inference; the detector
only needs one `person` class.

A frequent mistake is to load SAM3 bbox tracks into `getpixelvideo`, convert
each person to a single marker, then press **F9** (pose export). That produces
`nc: 1, names: ['object'], kpt_shape: [N, 3]` — **one** object per frame whose
keypoints are the N markers. A model trained that way predicts one box + N
keypoints and can never detect N separate people.

yolotrain now prints a warning and requires deliberate confirmation in the GUI.
CLI training is blocked unless `--allow-suspicious-pose` is supplied. Use that
override only for intentional keypoint datasets.

A high Box mAP does not prove this player task is correct. The wrong layout can
learn one large group box while Pose mAP remains very low.

Build the correct detection dataset directly from a SAM3 export:

```bash
uv run python -m vaila.sam_to_yolo build \
  --sam-tracks /path/processed_sam_*/<video>/sam_tracks.csv \
  --video /path/<video>.mp4 \
  --class-name person \
  --reuse-images-dir /path/vaila_dataset_*   # optional: reuse extracted frames
```

Then train with **Task=detect** (auto-detected, since there is no `kpt_shape`).
See `help/sam_to_yolo.md`.

The class **name** is metadata (labels store only the index `0`), so you can
change `object` → `person` at any point without touching label files:

```bash
# before training (dataset metadata)
uv run python -m vaila.sam_to_yolo rename-dataset --dataset DATASET --class-name person
# after training (baked into the .pt)
uv run python -m vaila.sam_to_yolo rename-model --weights best.pt --class-name person --output best_person.pt
```

The number of instances may vary per frame (no fixed count) — the tracker
handles however many `person` boxes the detector outputs.

## CLI

Open the unified local HTML guide:

```bash
uv run python -m vaila.yolotrain --guide
```

Open GUI:

```bash
uv run python -m vaila.yolotrain
```

Validate configuration without downloading weights or training:

```bash
uv run python -m vaila.yolotrain   --data /ABS/path/to/data.yaml   --task auto --model auto --device auto --dry-run
```

Train with YOLO26 auto task/model selection:

```bash
uv run python -m vaila.yolotrain \
  --data /ABS/path/to/data.yaml \
  --task auto --model yolo26m.pt \
  --epochs 100 --batch 8 --imgsz 1280 --device 0 \
  --name players_detect
```

Fine-tune from trained weights anywhere on disk:

```bash
uv run python -m vaila.yolotrain   --data /ABS/path/to/data.yaml   --task auto   --model /ABS/path/to/runs/yolo_training/weights/last.pt   --epochs 50 --device 0
```

Extra Ultralytics train arguments can be passed with repeated `--arg key=value`, for example:

```bash
uv run python -m vaila.yolotrain   --data /ABS/path/to/data.yaml   --arg workers=4 --arg cache=false
```

## Dataset Layouts

vailá/getpixelvideo layout:

```text
your_dataset/
├── data.yaml
├── classes.txt
├── train/images
├── train/labels
├── val/images
├── val/labels
└── test/images + test/labels (optional)
```

FIFA/Ultralytics layout:

```text
your_dataset/
├── data.yaml
├── images/train
├── labels/train
├── images/val
├── labels/val
└── images/test + labels/test (optional)
```
