Description

YOLO26-first trainer for datasets generated by getpixelvideo.py, SAM workflows loaded into getpixelvideo, and existing Ultralytics YOLO datasets.

Direct answer for player tracking: train a detect dataset with one bounding box per player. Keep GUI Task=auto only after confirming that data.yaml resolves to detect. Use pose only when every player has real body-joint labels. auto is dataset auto-detection; it is not a tracker.

Task guide: auto, detect, pose, or segment?

Think of each training label as an answer to “what should YOLO draw?”

  • auto: read data.yaml and labels, then choose a task. Safest only when the dataset is correct.
  • detect: draw one rectangle around each player. Best starting point for individual player tracking. Default model: yolo26m.pt.
  • pose: draw joints inside each object, such as head, shoulder, knee, and ankle for each person. Default model: yolo26m-pose.pt.
  • segment: color the exact silhouette/mask of each object. Default model: yolo26m-seg.pt.

Tracking happens after detection. BoT-SORT or ByteTrack receives player boxes and assigns IDs such as 1, 2, 3. The number of players is not the number of classes.

The CLI rejects known model/task mismatches early. For example, a dataset with kpt_shape is YOLO-pose, so --model yolo26x.pt is rejected and --model yolo26x-pose.pt is suggested.

Dataset guide: inspect before training

DatasetOne label row meansQuick sign
detectone player boxfive values: class, center x/y, width, height; no kpt_shape
poseone object box plus all keypoints belonging to that objectkpt_shape: [N, 3]
segmentone object polygon/maskmany normalized polygon coordinates

For a frame with 18 players, a detection label normally has 18 rows. A pose label for player body joints also has one row per player, with that player's joints after the box values.

Model guide

SizeUseTrade-off
n/squick tests, weaker GPUfastest, usually less accurate
mrecommended first serious runbalanced speed, memory, accuracy
l/xfinal experiments when time and VRAM allowslower and heavier; not automatically better on a weak dataset

The suffix must match the task: yolo26m.pt for detect, yolo26m-pose.pt for pose, and yolo26m-seg.pt for segment. For distant soccer players, start with yolo26m.pt, imgsz=1280, and a batch that fits memory (for example 4, 8, or -1 auto).

Maximum epochs

The GUI field is now named Maximum epochs and accepts values from 1 to 10000. It is a ceiling, not a promise: early stopping may finish before that value when validation stops improving.

  • 20 epochs: smoke test that verifies labels, memory, and losses.
  • 100 epochs: reasonable first baseline.
  • 200+ epochs: use only while validation metrics still improve; more epochs cannot repair wrong labels.

GUI: step by step

  1. Open YOLO + SAM → Train YOLOv26.
  2. Select the dataset folder or data.yaml.
  3. Read the colored Task explanation. For player tracking it should say DETECT.
  4. Keep Model=auto or start with yolo26m.pt.
  5. Set Maximum epochs, Batch, Image size, Device, and Run name.
  6. Start training. Suspicious players-as-keypoints pose datasets now show a blocking confirmation.

Single Help button (v0.3.65): the GUI has one Help button (instead of separate Help / Open HTML / Diagnostics / Model Guide / Task Guide buttons). It opens an in-app guide window that also contains Open full guide in browser (HTML) and Print diagnostics + CLI command buttons; the same quick guide is printed to the terminal.

STOP in the terminal (v0.3.65): the suspicious-pose gate now prints a clear STOP block with the sam_to_yolo repair command to the terminal, not only the GUI dialog.

Temporal split: datasets exported from getpixelvideo / sam_to_yolo default to a chronological train/val/test split, which avoids the inflated mAP caused by a random split leaking near-duplicate video frames between train and val.

CLI: same work as the GUI

Open this guide from the terminal:

uv run python -m vaila.yolotrain --guide

Validate first; this downloads nothing and starts no training:

uv run python -m vaila.yolotrain \
  --data /ABS/path/to/data.yaml --task auto --model auto --dry-run

Train a player detector:

uv run python -m vaila.yolotrain \
  --data /ABS/path/to/data.yaml \
  --task auto --model yolo26m.pt \
  --epochs 100 --batch 8 --imgsz 1280 --device 0 \
  --name players_detect

Fine-tune from trained weights:

uv run python -m vaila.yolotrain \
  --data /ABS/path/to/data.yaml \
  --model /ABS/path/to/weights/last.pt \
  --epochs 50 --device 0

Layouts

Supports train/images + train/labels and FIFA/Ultralytics images/train + labels/train layouts.

How to know training is running (v0.3.57)

Watch the terminal that launched vailá after Start Training:

  • >> yolotrain: Preparing dataset cache... — first phase; 10k+ images can take minutes.
  • Ultralytics tqdm bars on stderr during each epoch.
  • >> yolotrain: Epoch N/100 complete | train_loss=... | mAP50=... after every epoch.
  • GUI console shows epoch summaries; tqdm stays in the terminal.

Outputs: <dataset>/runs/yolo_training/weights/best.pt and last.pt.

Class name object vs person

names: ['object'] vs ['person'] is label metadata only. Pose datasets (kpt_shape) use yolo26*-pose.pt. Person detection needs bbox-only detect labels, names: ['person'], task detect, and yolo26m.pt. Continue training with Browse Weights or --model /path/to/weights/last.pt.

Player tracking: correct SAM → YOLO workflow

To detect/track N people or objects you need a detection dataset: one bounding box per instance per frame, a single class (e.g. person), and no kpt_shape. The number of people is not the number of classes — the tracker (BoT-SORT/ByteTrack) assigns IDs 1..N at inference.

A frequent mistake: load SAM3 bbox tracks into getpixelvideo, convert each person to one marker, then press F9 (pose export). That yields nc: 1, names: ['person'], kpt_shape: [N, 3] — one object per frame whose keypoints are the players. It can never detect N separate people. The GUI now asks for explicit confirmation; CLI training is blocked unless --allow-suspicious-pose is deliberately supplied.

A high box score does not prove the player task is correct. In this wrong layout, YOLO can learn one large group box and report high Box mAP while Pose mAP remains very low. Inspect what one label row represents before trusting metrics.

Build the correct detection dataset from a SAM3 export:

uv run python -m vaila.sam_to_yolo build \
  --sam-tracks /path/sam_tracks.csv \
  --video /path/video.mp4 \
  --class-name person \
  --output /path/player_detect_dataset

Then validate and train. Task=auto should resolve to detect because the new data.yaml has no kpt_shape.

Diagnostics

The GUI prints resolved train/val paths, image counts, label counts, detected task, class names, candidate YAML files, and full tracebacks for YAML or training errors to the launching terminal.
