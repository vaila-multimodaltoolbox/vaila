Sapiens2 Pose (vaila_sapiens.py)

Version: 0.3.76  |  Updated: 2026-07-07

GUI: Frame B → YOLO + FB → Sapiens2 Pose

Category: Markerless 2D / Meta (Facebook)

Top-down 308-keypoint human pose on video using Meta's DETR person detector + Sapiens2 ViT pose head.

License: Sapiens2 weights use Meta's Sapiens2 License (not AGPL). Accept the Hugging Face model terms before downloading.

Requirements

  • NVIDIA CUDA — no CPU or macOS path in this integration.
  • During install_vaila_linux.sh with GPU support, answer Y to install optional Sapiens2 Pose.

Setup (first time)

uv sync --extra sapiens
bash bin/setup_sapiens2.sh

This clones .local/third_party/sapiens2/ and downloads weights into vaila/models/sapiens2/.

GUI — how to use

  1. Open vailá → YOLO + FB → Sapiens2 Pose
  2. Dir… = batch all videos in a folder; File… = single video
  3. Choose the output parent directory
  4. Model default 1b; stride 1 = every frame
  5. Detection & keypoint thresholds — --bbox-thr, --nms-thr, --kpt-thr, --max-persons
  6. GPU & advanced — --device, --pose-batch-size (empty = auto), --flip-test, overlay on/off
  7. Click Run — terminal prints full >> Equivalent CLI with --output-base

On Run, the terminal prints >> vaila/vaila_sapiens: Equivalent CLI with every flag you chose (copy/paste to reproduce without the GUI).

CLI flags — what each option does

Input / output

FlagDefaultWhat it isWhat changes when you use it
-i / --input — Path to one .mp4 (or similar) or a folder of videos. Single file = one run. Folder = batch (skips *_sapiens_overlay.* and processed_sapiens_* subfolders).
-o / --output — Parent directory where results are written. Creates processed_sapiens_YYYYMMDD_HHMMSS/<video_stem>/ with CSVs, JSON and overlay.

Model and inference

FlagDefaultWhat it isWhat changes when you use it
--model 1b Sapiens2 pose checkpoint size: 0.4b, 0.8b, 1b, 5b. Larger = better quality, more VRAM and slower. 1b is the default for RTX 4090 24 GiB. Use the same --model for --download-weights and inference.
--stride 1 Run the pose network every N video frames. 1 = every frame (slowest, smoothest). 2 or 3 = fewer GPU passes; skipped frames reuse the nearest inferred pose in overlay and CSVs. Use the same stride on all cameras in a multi-view study.
--device 0 CUDA GPU index (0 = first NVIDIA card). Pick another index if you have multiple GPUs and want to leave GPU 0 free.
--flip-test off Optional left–right flip ensemble inside the pose head. Slightly better accuracy, but about 2× VRAM. Off by default on 24 GiB cards.
--pose-batch-size auto How many person crops the pose head runs at once per frame (see below). Auto: 1 (5b), 2 (1b), 4 (0.4b/0.8b). Lower (e.g. 1) if CUDA OOM on crowded frames.

What is --pose-batch-size?

Sapiens2 in vailá is top-down pose: it does not estimate all joints on the full frame in one shot. Each video frame goes through two steps:

  1. Person detection (DETR) — finds people and draws a bounding box around each one (controlled by --bbox-thr, --nms-thr, --max-persons).
  2. Pose estimation (Sapiens2 ViT) — for each box, crops that person from the frame and predicts 308 keypoints on the crop.

--pose-batch-size controls step 2 only: how many person crops are sent through the pose network in a single GPU forward pass on the same frame.

Example: a frame with 12 detected persons and --pose-batch-size 4 means the pose head runs 3 passes on that frame (4 + 4 + 4 crops). With --pose-batch-size 1, it runs 12 passes — slower, but uses much less VRAM per frame.

Auto (default): if you omit the flag (GUI field empty), vailá picks a safe default from the model size:

  • 5b → batch 1 (largest weights; least parallel crops)
  • 1b → batch 2
  • 0.4b / 0.8b → batch 4

When to lower it: if the process crashes with CUDA out of memory (OOM) on crowded scenes (many players, audience, beach, etc.), try --pose-batch-size 1. You can also reduce --max-persons to cap how many boxes DETR keeps, but --pose-batch-size is the direct knob for VRAM during pose inference.

Not the same as --stride: stride skips frames; pose-batch-size batches persons within one frame.

Detection thresholds

FlagDefaultWhat it isWhat changes when you use it
--bbox-thr 0.3 Minimum DETR person-detection score to keep a bounding box. Higher = fewer persons detected (misses faint/small people). Lower = more boxes, more false positives.
--nms-thr 0.3 Non-maximum suppression overlap between person boxes. Lower = stricter deduplication (one box per person). Higher = may keep overlapping duplicate boxes.
--max-persons 8 Max persons per frame (top scores kept). Lower if VRAM is tight on crowded scenes; extra detections are dropped.

--kpt-thr — keypoint confidence cutoff

Each of the 308 joints gets a confidence score from the pose model (0 = unreliable, 1 = very confident). --kpt-thr is the minimum score for a joint to be treated as visible in post-processing. Default 0.3 follows the usual MMPose / Sapiens convention.

FlagDefaultWhat it isWhat changes when you use it
--kpt-thr 0.3 Per-joint confidence cutoff (0.0–1.0). See table below — effect depends on which output file you use.

Where --kpt-thr applies:

  • Overlay MP4 — joints with score < threshold are not drawn (skeleton looks cleaner).
  • <stem>_id_NN_sapiens_pose.csv and <stem>_getpixelvideo_pose.csv — low-confidence joints are written as empty cells (getpixelvideo shows gaps instead of noisy points).
  • sapiens_points.csv — mid-hip (pN_hx, pN_hy) is averaged only from hip keypoints above the threshold; if none pass, hip cells are empty. Foot and bbox-center columns use the detection box, not this threshold.

Where --kpt-thr does not filter:

  • <stem>_sapiens_vaila.csv — long format keeps all joints with raw score column; you can filter later in pandas/Excel.
  • <stem>_markers.csv — foot point comes from the bbox bottom anchor, not from joint confidence.
  • sapiens_vaila_*.csv bbox anchors — derived from the person box, not per-joint scores.
  • <stem>_predictions.json — raw model output with every score preserved.

Practical tuning:

  • 0.1–0.2 — keep more joints; useful when limbs are often occluded but expect noisier points in overlay and getpixelvideo.
  • 0.3 — balanced default for sports / broadcast video.
  • 0.4–0.6 — stricter; skeleton and wide CSVs show only high-confidence joints (more gaps, less jitter).

The value used is saved in <stem>_predictions.json as kpt_thr_used.

Run control

FlagDefaultWhat it isWhat changes when you use it
--no-overlay off Skip writing the skeleton preview video. Faster run and less disk use; all CSV/JSON outputs are still written.
--dry-run off List videos and settings without loading the model. Checks paths, model name and stride; no GPU inference, no output files.
--download-weights off Download pose checkpoint + shared DETR detector from Hugging Face. Writes to vaila/models/sapiens2/ for the --model you pass; exits without processing video.
--quiet off Minimal terminal output. Hides the tqdm frame bar and periodic heartbeat lines. GUI batch enables this automatically.
--open-help off Open this help page in the browser. Exits immediately after opening vaila/help/vaila_sapiens.html.

Terminal progress

CLI runs print >> vaila/vaila_sapiens: status lines plus a tqdm frame bar on stderr. The first 1–2 minutes are often model load before the bar moves — that is normal. Use --quiet to suppress it (GUI batch does this automatically).

Output directory

Each run creates one folder under the output parent:

<output_parent>/processed_sapiens_YYYYMMDD_HHMMSS/<video_stem>/

Subprocess-per-video isolation reuses the parent --output-base (no second empty folder).

getpixelvideo workflow

Full skeleton (308 keypoints):

  1. Load → <stem>_id_01_sapiens_pose.csv (or <stem>_getpixelvideo_pose.csv for single person)
  2. Or Load Tracking CSV → <stem>_sapiens_vaila.csv → enter person_id when prompted

Multi-person bboxes:

  1. Load Tracking CSV → sapiens_bbox_tracks.csv → press Enter at anchor prompt (keep boxes)

REC2D/REC3D uses <stem>_markers.csv (one foot point per person).

Biomechanics workflow

  1. Run Sapiens2 on each camera view (same --stride across views for multi-camera).
  2. Feed <stem>_markers.csv into Rec2D / Rec3D with your DLT parameters.
  3. Use <stem>_sapiens_vaila.csv for joint-level 2D kinematics (308 keypoints).
  4. Optional: load wide pose CSVs or sapiens_bbox_tracks.csv in getpixelvideo for QA/editing.

VRAM tips (RTX 4090)

  • Flip-test is off by default (doubles VRAM). Use --flip-test only when you need max accuracy.
  • On 1080p multi-person with 5b, try --stride 2 or higher to avoid OOM.
  • Each video runs in an isolated subprocess — a failed run does not affect the next one.
